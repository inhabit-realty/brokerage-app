A

Authentication

API Key for agent-level access, OAuth 2.0 for multi-user portal auth

Required First
i The portal will use OAuth 2.0 so each agent authenticates with their own FUB account. Register your system at docs.followupboss.com and include X-System and X-System-Key headers in every request.
API Key (Per-Agent)
Each FUB user has a unique API key under Admin → API. Used for server-side integrations. Pass as HTTP Basic Auth username with blank password.
# Every request must include system headers GET https://api.followupboss.com/v1/people Authorization: Basic base64("API_KEY:") X-System: "InhabitPortal" X-System-Key: "your_system_key"
OAuth 2.0 (Portal Standard)
Recommended for the PWA. Agents click "Connect FUB", redirected to FUB consent screen, return with auth code, exchange for access + refresh tokens.
# Step 1 redirect to consent GET https://app.followupboss.com/oauth/authorize ?response_type=auth_code &client_id=YOUR_CLIENT_ID &redirect_uri=https://portal.inhabit.com/auth/fub # Step 2 exchange code POST https://app.followupboss.com/oauth/token grant_type=authorization_code &code={auth_code} # Step 3 refresh POST https://app.followupboss.com/oauth/token grant_type=refresh_token &refresh_token={token}

People (Contacts)

The core resource — every lead, client, and contact in FUB

Core Resource
MethodEndpointDescription
GET/peopleList all contacts. Filter by stage, tag, assigned agent, date ranges. Supports full-text search.
POST/peopleCreate a new contact. FUB auto-deduplicates by email/phone.
GET/people/:idFull contact record with all fields, tags, stage, assigned agent, custom fields.
PUT/people/:idUpdate any contact field. Partial updates supported.
DEL/people/:idArchive/delete a contact. Soft delete — contact stays as deleted.
GET/people/checkDuplicateCheck if a person exists before creating. Pass email or phone.
GET/people/unclaimedLeads not yet assigned to an agent.
POST/people/claimAgent claims an unclaimed lead.
POST/personAttachmentsAttach files to a contact record.
E

Events (Lead Intake)

Preferred way to push leads from MLS VOW, website, or any source into FUB

Critical for MLS VOW
! When a buyer registers on the MLS VOW search portal, POST to /events. FUB will deduplicate, assign based on rules, and fire action plans automatically. This is the correct intake pattern — not /people POST.
MethodEndpointDescription
POST/eventsSend a lead event. FUB deduplicates, assigns, triggers automation.
GET/eventsList lead events with source, type, person data.
GET/events/:idGet a specific event and the person it created or matched.
